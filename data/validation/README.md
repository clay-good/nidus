# Machine pre-verification

This directory holds a **machine-assisted pre-verification** layer for the
dataset. It exists to make human verification faster, not to replace it.

## What it is (and isn't)

Nidus's value rests on parameters being checked against their source papers.
Most parameters are still `unverified` — the central value comes from the
literature, but no human has confirmed it against the original. Reading 200+
papers from scratch is the bottleneck.

This layer does the legwork a machine *can* do honestly:

1. For each citation, fetch the best available source text from
   [Europe PMC](https://europepmc.org/) — open-access full text where it
   exists, otherwise the abstract.
2. For each parameter, compare the stored value against what that source
   actually reports, and record a verdict **with a verbatim supporting
   quote** so a human can confirm in seconds.

It **never** sets `extraction.review_status`. Promoting a parameter to
`verified` (or `contested`) is a human action after reading the source.
Think of this as a pre-filled worklist, not a verdict.

## The artifacts

| File | What it is |
| ---- | ---------- |
| [`REVIEW_QUEUE.md`](REVIEW_QUEUE.md) | Human worklist, grouped by verdict (mismatches first), each with the quote and what to confirm. **Start here.** |
| `machine_checks.json` | The same data, structured, one record per parameter. |
| `sources/` | Cached source text (gitignored — publisher abstract/full-text content). |
| `checks/` | Per-parameter check records (gitignored staging; regenerated by the pipeline). |

## Verdicts

| Verdict | Meaning |
| ------- | ------- |
| `match` | The source explicitly states a value consistent with the stored one (quote attached). |
| `close` | The source supports the magnitude/direction but the exact figure or statistic differs, or gives a range the stored value falls in (quote attached). |
| `mismatch` | The source appears to report a *different* value (quote attached) — highest priority. |
| `not_found` | The source was fetched but the specific value isn't in the available text (usually it lives in a table/figure, or only the abstract was available). |
| `source_unavailable` | No source text could be retrieved (book, pre-index paper, or no abstract). |

A `match`/`close`/`mismatch` verdict **must** carry a verbatim quote; if there
is no quote, the verdict is `not_found`. The machine is deliberately
conservative — when unsure it reports `not_found`, never `match`.

## How a reviewer uses it

1. Open `REVIEW_QUEUE.md`. Work top-down: mismatches, then Tier A/B.
2. For a given parameter, read the quote. If it settles the value, you may
   still want the full paper for context; if the verdict is `not_found` /
   `source_unavailable`, you'll need the full PDF or a library copy.
3. Confirm or correct the value in `dataset/parameters/<subsystem>.json`, set
   `extraction.review_status` to `verified` (or `contested`), fill
   `extraction.reviewer` and `extraction.date`, and improve `tier_rationale`
   if needed.

## Regenerating

```bash
python scripts/fetch_sources.py          # cache source text from Europe PMC
# (comparison pass writes per-parameter records into checks/)
python scripts/build_review_queue.py     # aggregate -> machine_checks.json + REVIEW_QUEUE.md
```

The comparison pass that writes `checks/` is run by an LLM reading each
source against each stored value under the rules above; `machine_checks.json`
is the durable record of its output.

## Coverage (at generation time)

243 parameters: 11 `match`, 22 `close`, 4 `mismatch`, 137 `not_found`,
69 `source_unavailable`. So ~72% of parameters have *some* fetchable source
evidence; the specific stored number is quote-confirmable for ~33 of them
today, and 4 stored values look wrong against their source and should be
checked first. The rest need full-text/table access or a library copy — which
is exactly the map a human reviewer needs to spend their time well.
