# Release runbook

Everything in the repository is built, tested, and release-ready. What remains
to actually *launch* nidus is a set of **maintainer actions** that require
external accounts (PyPI, Zenodo, Streamlit Cloud), a tagged release, or
third-party submissions. None of these can be performed by the codebase or an
automated agent — they are listed here so each is a single, well-defined step.

The repo automation does the heavy lifting: pushing a `v*.*.*` tag publishes to
PyPI and cuts a GitHub release; every push to `main` already redeploys the docs
site to the `gh-pages` branch.

## One-time setup (do once, ever)

| Step | Where | Notes |
| ---- | ----- | ----- |
| Reserve the `nidus` project + configure a **PyPI Trusted Publisher** | pypi.org → project settings | Environment `pypi`, workflow `release.yml`. No API token needed — `release.yml` uses OIDC. |
| Enable **GitHub Pages** | repo Settings → Pages → Source → *Deploy from a branch* → `gh-pages` | `docs.yml` already pushes the built site to `gh-pages` on every `main` push; this just exposes it. |
| Connect **Zenodo ↔ GitHub** | zenodo.org → GitHub → flip the `nidus` repo *on* | Zenodo then mints a DOI automatically when a GitHub release is published. |
| Create the **Streamlit Cloud** app | share.streamlit.io → New app → this repo, main file `dashboard/app.py` | Pre-PyPI, add `git+https://github.com/clay-good/nidus.git#subdirectory=python` to `dashboard/requirements.txt` (or rely on the published `nidus` once it is on PyPI). |

## Per release (e.g. `v0.3.0`)

1. Drop the `.dev0` suffix in [`python/pyproject.toml`](python/pyproject.toml) (`version = "0.3.0"`).
2. Move the `[Unreleased]` entries in [`CHANGELOG.md`](CHANGELOG.md) and
   [`dataset/CHANGELOG.md`](dataset/CHANGELOG.md) under a dated `## [0.3.0]` heading.
3. Commit and push to `main`. Confirm CI is green.
4. Tag and push:
   ```bash
   git tag -a v0.3.0 -m "nidus v0.3.0"
   git push origin v0.3.0
   ```
   This triggers [`release.yml`](.github/workflows/release.yml): builds the
   wheel + sdist, publishes to PyPI (OIDC), and cuts a GitHub release with
   auto-generated notes. The release publish triggers the Zenodo deposit.

## Post release

1. Copy the minted **Zenodo concept DOI** into [`CITATION.cff`](CITATION.cff),
   the README badge, and the essay's "Where to find it" line (it currently uses
   a placeholder DOI).
2. Tick the now-satisfied release boxes in the specs:
   - `docs/specs/v0.3/01-dataset-and-dashboard.md` — Streamlit URL, Zenodo DOI.
   - `docs/specs/v0.3/03-outreach-and-essay.md` — Pages serving, About/topics.
   - `docs/specs/v0.4/01-mechanistic-modeling-interop.md` — Zenodo deposit.

## Outreach & external model repositories (optional, after launch)

These are launch/visibility actions, not build steps. The artifacts they
submit are all already generated by `nidus export --format ...`.

- **Outreach:** post the essay to Twitter/Mastodon/Bluesky, r/bioinformatics,
  r/datasets, Show HN, Data Is Plural; email a handful of perinatal researchers
  whose papers the dataset cites; set the repo About/topics to point at the essay.
- **BioModels Database:** submit `exports/sbml/nidus_pregnancy.omex` (or the
  composed SBML). Accession arrives on the curators' timeline.
- **Physiome Model Repository:** create a workspace exposing the CellML 2.0 and
  1.1 exports.
- **PhysioCell:** open a PR to `MathCancer/PhysiCell-tutorials` with
  `docs/examples/physicell_placental_villous/`.
- **Deferred to post-v0.4:** BioPortal ontology submission, JOSS paper.

## Regenerating the exports a submission needs

```bash
pip install -e "python[export]"
nidus export --format sbml       --output exports/sbml/
nidus export --format cellml     --output exports/cellml/
nidus export --format cellml     --output exports/cellml-1.1/ --cellml-version 1.1
nidus export --format physiocell --output exports/physiocell/
nidus export --format composed   --output exports/composed/
nidus export --format omex       --output exports/nidus_pregnancy.omex
```

(CI also regenerates these on every dataset change and uploads them as the
`nidus-exports` artifact.)
