# PhysiCell placental villous slice — minimal example

A small 2D PhysiCell project that uses nidus-derived parameters to seed a
slice through a placental villous tree. Maternal blood diffuses oxygen
and glucose into a microvillous boundary; trophoblast cells consume
oxygen and shuttle glucose into the fetal compartment via GLUT1/GLUT3
Michaelis–Menten kinetics.

The whole point of this example is that **every numerical input comes
from nidus**: no magic numbers in the XML, every parameter carries a
citation key + confidence tier in the `description` attribute, and the
mathematical kinetics match the SBML / CellML exports byte-for-byte.

## What you get

```
physicell_placental_villous/
├── README.md                    ← you are here
├── PhysiCell_settings.xml       ← BioFVM domain + cell defs + user_parameters
├── nidus-parameters.xml         ← drop-in <user_parameters/> generated by `nidus export`
├── custom_modules/
│   ├── custom.cpp               ← C++ stub showing how to read nidus params
│   └── custom.h
└── tutorial.ipynb               ← walkthrough: regenerate params, build, simulate
```

`nidus-parameters.xml` is the **canonical** parameter source. It is
generated from the bundled dataset by:

```bash
nidus export --format physiocell --output docs/examples/physicell_placental_villous/
```

`PhysiCell_settings.xml` `<include>`s that file by convention; the
maintainer regenerates it whenever the upstream dataset changes.

## Why this matters

PhysiCell projects historically hardcode dozens of biologically
load-bearing constants inline. Reviewers (and future-you) have no way to
trace them back to literature. Wiring nidus into the loop means:

1. Every Km/Vmax/P50/baseline value comes with a citation key and a
   confidence tier in the same file the simulator reads.
2. When a citation is challenged, you grep the tag, update the dataset
   parameter, re-run `nidus export`, re-run the simulation. No
   archaeology.
3. The mechanistic kinetics in `custom.cpp` use the same algebraic form
   as the SBML/CellML exports, so cross-platform reproductions remain
   consistent.

## Running it

1. Install PhysiCell (https://github.com/MathCancer/PhysiCell). This
   example targets PhysiCell ≥ 1.13.
2. Place this directory under `PhysiCell/sample_projects/`.
3. From the PhysiCell root:
   ```
   make load-project PROJECT=physicell_placental_villous
   make
   ./project ./sample_projects/physicell_placental_villous/PhysiCell_settings.xml
   ```
4. The simulation writes output `.mat` snapshots to
   `output/`; visualise with PhysiCell's `pyMCDS` reader.

The default settings simulate a ~150 µm × 150 µm slice for 6 hours of
biological time; total runtime is < 1 minute on a laptop.

## Provenance audit checklist

When you reuse this in a paper, confirm:

- [ ] `nidus-parameters.xml` was generated from a published nidus tag
      (e.g. `v0.4.0`) rather than a dirty working copy.
- [ ] The simulation reads `glucose_glut1_*` / `glucose_glut3_*` /
      `gas_max_equilibration` / `o2_hb_p50_maternal` etc. by id (not
      hardcoded copies).
- [ ] Any parameter overrides for sensitivity analysis are recorded in
      a separate XML so the baseline is preserved.

See [docs/specs/v0.4/01-mechanistic-modeling-interop.md] for the full
interop contract and round-trip validation strategy.
